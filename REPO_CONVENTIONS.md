# Repository Conventions

How to organize files in this repo. Follow these rules for any new content you add so the layout stays predictable for the team and for AI assistants.

## Folder Layout Rule: `src/` for Sources, Parent for Outputs

Every folder that contains both **source files** (editable inputs) and **rendered outputs** (final artifacts the team or evaluators look at) splits them as follows:

```
some_folder/
├── src/                          ← all source files live here
│   ├── document.xlsx             ← editable source
│   ├── document.tex              ← editable source
│   ├── images/                   ← image assets used by sources
│   └── helper.py                 ← scripts that generate other sources
└── document.pdf                  ← rendered output, sits at parent level
```

### Why this convention

- The output PDF is the deliverable, so it should be the most visible file in the folder.
- Source files cluster together in `src/` and aren't mixed with renders or standalone documents.
- A reader looking for "the final report" or "the latest BOM" lands on a PDF immediately.
- A contributor editing content goes one level into `src/`.

## What Counts as a Source

Put these in `src/`:

| File type | Examples |
|---|---|
| Spreadsheets (editable) | `.xlsx`, `.csv` |
| LaTeX documents | `.tex`, `.bib`, `.sty`, `.cls` |
| Markdown documents that get compiled | `.md` (only if rendered to PDF) |
| Images referenced by `.tex` or `.md` builds | `.png`, `.jpg`, `.svg` under `src/images/` |
| Build / generation scripts | `.py`, `.sh`, `.js` that emit other files |
| Project files for proprietary editors | `.dmd` (Do-more Designer), `.dwg`/`.aepx`/`.wdp` (AutoCAD Electrical) |

## What Counts as an Output

Put these at the parent folder level (NOT inside `src/`):

| File type | Examples |
|---|---|
| Final rendered PDF | `document.pdf` |
| Plotted or exported image used as a final deliverable | (rare) |

## What Stays at the Parent Level (Neither Source nor Output)

Some files are standalone documents — they're neither edited locally nor generated by a build. Leave them at the parent level alongside the rendered outputs:

| File type | Examples | Why |
|---|---|---|
| Reference PDFs (not authored by team) | Vendor datasheets, sponsor proposals, course handouts | Read-only |
| Standalone markdown documentation | `README.md`, `*_Wiring.md`, `Programmed_Parameters.md` | Read directly on GitHub |
| Standalone .docx files | Signed contracts, ethics forms | Final-form documents |
| Reference workbooks (not editable sources) | _none currently_ | |

## When Adding New Content

### Authoring a new report or document

1. Create your source file in `<folder>/src/`.
2. If your source uses images, put them in `<folder>/src/images/`.
3. Reference images from your tex/md with the path `images/foo.png` (relative to your source file's folder).
4. Compile / render the output PDF.
5. Place the output PDF at `<folder>/` (one level above `src/`).

### Creating a new top-level folder

When you make a new top-level folder (for example `safety/`), follow the same rule:

```
safety/
├── new_doc.pdf
└── src/
    ├── new_doc.tex
    └── images/
```

### Updating an existing source

1. Edit the file in `<folder>/src/`.
2. Recompile or re-render to produce the updated output.
3. Replace the output PDF at `<folder>/`.
4. Commit both the changed source AND the updated output PDF in the same commit so they stay in sync.

## What NOT to Do

- Don't put output PDFs inside `src/`. They belong at the parent level.
- Don't put source files (xlsx, tex) at the parent level when there's also a rendered output. Move them to `src/`.
- Don't create a `src/` subfolder if a folder only contains standalone documentation (markdown, vendor PDFs, signed contracts). For example, `VFD/` has many standalone .md docs that stay at the `VFD/` level — only the `VFD_Decision_Matrix.xlsx` (which has a rendered PDF counterpart) needs to live in `VFD/src/`.
- Don't generate output PDFs into `src/` and leave them there. Move them up one level.
